# Phase B-1: AI 코드 생성 파일별 분리 — 완료 보고서
> 작성: 자비스 (세션 7, 2026-03-26)

---

## 배경 — 왜 이걸 했는가

현재 AI가 한 페이지에 대해 **여러 파일(page.tsx + 컴포넌트들)을 한꺼번에 생성** → 긴 응답이 토큰 제한에 걸려 **잘림** → F4(이어붙이기)가 시도 → **시간 낭비 + 품질 저하**

특히 flash(Haiku) 모델은 maxTokens 8192 → 복잡한 페이지는 거의 100% 잘림.

---

## 뭘 했는가

### 핵심: Step 3를 **페이지별 루프 → 파일별 개별 생성 루프**로 변경

**변경 전** (ai.service.ts Step 3):
```
for (const page of pages) {
  AI에게: "이 페이지의 모든 파일을 생성해"
  → AI가 page.tsx + 컴포넌트 3~5개를 한 응답에 생성
  → 긴 파일 잘림 → F4 이어붙이기
}
```

**변경 후**:
```
1. 아키텍처에서 파일 목록 평탄화 (FileTask[])
2. for (const task of fileTasks) {
     AI에게: "이 파일 1개만 생성해!"
     → 이미 생성된 파일 목록을 컨텍스트로 전달
     → 잘림 없음, F4 불필요
   }
```

### 파일 분리 전략

| 컴포넌트 수 | 전략 | 예시 |
|-------------|------|------|
| 0~3개 | page.tsx 1파일에 모두 포함 | 단순 페이지, 설정 등 |
| 4개 이상 | page.tsx (핵심 2개 인라인) + 나머지 컴포넌트 2개씩 묶어 별도 파일 | 대시보드, 복잡한 CRM |

### import 일관성 보장

- 이미 생성된 파일 목록(최근 5개)을 프롬프트에 포함
- AI가 `import { XxxComponent } from './components/xxx'` 형태로 참조 가능
- 컴포넌트 파일에는 `export`를 명시하도록 프롬프트에 지시

### 프롬프트 분리

- **페이지 파일 (page.tsx)**: Supabase 스키마 + CRUD 패턴 + 스마트 분석 컨텍스트 포함
- **컴포넌트 파일**: 소속 페이지명 + export 규칙 중심, 스키마는 최소한만

## 기대 효과

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 파일 잘림 | 복잡한 페이지에서 빈번 | 거의 없음 (1파일 = 짧은 응답) |
| F4 이어붙이기 | 자주 발동 (시간 낭비) | 거의 불필요 |
| 코드 품질 | 잘린 코드 → 불완전 | 완전한 파일 단위 생성 |
| 생성 시간 | 10페이지 × 1호출 = 10회 | 10~15파일 × 1호출 = 10~15회 (약간 증가) |
| import 일관성 | 보장 안 됨 | 이미 생성 파일 컨텍스트 전달 |

### 트레이드오프
- **API 호출 수 약간 증가** (10회 → 10~15회): 복잡한 페이지에서 컴포넌트 분리 시
- **하지만**: F4 이어붙이기 호출 감소 + 빌드 실패 감소 → 총 비용은 비슷하거나 감소
