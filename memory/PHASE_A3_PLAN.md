## 왜 하는가

A-1에서 비주얼 에디터를 만들고, A-2에서 저장/배포 분리 + 치환 리스크를 해결했다.
하지만 아직 **조용히 실패하는 버그**가 남아있다.

### 핵심 문제: 치환 실패를 사용자가 모름

A-2에서 openingTag로 맥락 치환을 했지만, DOM과 소스코드는 미세하게 다를 수 있다:
```
소스코드: <h1 className="text-blue-500">
브라우저: <h1 class="text-blue-500">
              ↑ className vs class 차이!
```
속성 순서, 띄어쓰기, self-closing 태그 등도 다를 수 있다.

치환이 실패하면:
- 사용자: "색상 바꿨는데?" → DOM에서는 바뀌어 보임 → [수정사항 적용] → 배포됨
- 근데 실제 소스코드는 안 바뀜 → 배포된 사이트에서 원래대로
- 사용자: "방금 바꿨는데 왜 또 원래대로야??" → 원인 모름

**조용히 실패하는 버그가 가장 위험하다.**

## 🔴 1순위: 치환 실패 감지 + 사용자 알림

### 구현
- inline-edit API 응답에 결과 추가:
  ```
  성공: { success: true, matchFound: true }
  실패: { success: false, matchFound: false, reason: "매칭 대상 못 찾음" }
  ```
- project.service.ts inlineEdit 메서드: replace 후 실제로 바뀌었는지 확인 (before === after면 실패)
- InlineEditor.tsx: API 응답 확인 → 실패 시 "⚠️ 수정을 적용하지 못했습니다" 토스트/알림 표시
- 실패 시 DOM 변경도 롤백 (또는 "소스코드에 반영되지 않았습니다. AI 수정을 이용해주세요" 안내)

### 왜 중요한가
- 사용자가 실패를 인지해야 다른 방법(AI 수정)을 시도할 수 있음
- 조용히 실패하면 "배포했는데 안 바뀌었어요" 문의 폭주
- 디버깅도 쉬워짐 (로그에 실패 기록 남음)

## 🟠 2순위: 색상 변경 디바운스

### 문제
컬러피커에서 색상을 드래그하면 onChange가 초당 수십 번 발생 → 매번 API 호출 → 서버 폭격

### 구현
- InlineEditor.tsx 색상 변경 부분에 디바운스 추가
- DOM 변경(postMessage)은 즉시 (사용자 눈에 바로 반영)
- DB 저장(API 호출)은 500ms 디바운스 (마지막 변경 후 500ms 뒤에 1번만 호출)
- 간단한 setTimeout + clearTimeout으로 구현. 라이브러리 불필요.

### ⚠️ 경쟁 상태 방지 (중요!)
- 디바운스 대기 중(API 저장이 아직 안 날아감)에 사용자가 [수정사항 적용] 버튼을 누르면 → 저장 안 된 채로 배포 시작 → 방금 바꾼 색상 날아감!
- 해결: 디바운스 대기 중일 때 [수정사항 적용] 버튼을 disabled 처리 또는 "저장 중..." 상태로 표시
- page.tsx에 isSaving state 추가 → 디바운스 시작 시 true, API 완료 시 false
- [수정사항 적용] 버튼: isSaving이면 disabled

## 🟡 3순위: 변경 건수 뱃지

### 구현
- page.tsx: hasUnsavedChanges를 boolean → number(unsavedCount)로 변경
- 인라인 편집 성공 시 unsavedCount++
- [수정사항 적용] 버튼에 "수정사항 적용 (3)" 형태로 건수 표시
- 배포 완료 시 unsavedCount = 0
- 사용자가 "내가 몇 개 바꿨는지" 한눈에 알 수 있음
