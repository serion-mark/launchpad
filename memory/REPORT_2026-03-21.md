**파일**: `api/src/project/deploy.service.ts:348`

```typescript
// 현재 (버그)
const errorFileRegex = /(?:\.\/|src\/|pages\/)([^\s:]+\.(?:tsx?|jsx?))/g;

// 수정 필요

```

**증상**: `./app/sales/page.tsx:69:58` 에러가 나왔는데, 정규식이 `app/` 패턴을 지원하지 않아 에러 파일을 찾지 못함. 결과: 엉뚱한 파일 5개를 AI에 보내고 같은 에러 3회 반복.

### 🟡 BUG-2: F6 동일 에러 반복 감지 없음

**현재**: 매 시도마다 에러 로그를 새로 파싱하지만, **같은 파일에서 같은 에러가 반복되는지 확인하지 않음**
**필요**: 이전 시도와 동일한 에러면 → 해당 파일을 직접 generatedCode에서 찾아 AI에 전달

### 🟡 BUG-3: F4 이어서 생성이 프론트엔드 페이지 생성 직후 실행 안 됨

**현재**: `generateFullApp()`에서 F4 continueGeneration은 **모든 파일 생성 완료 후** 일괄 실행
**문제**: `sales/page.tsx`가 69줄에서 잘렸는데 continueGeneration이 안 되었을 가능성
**확인 필요**: 서버 로그에서 `[F4 코드 잘림 감지]` 로그가 찍혔는지 확인

## 3. 구조적 개선사항

### S1. F6 자동수정 한계 — "에러 파일 특정" 정확도

현재 F6은 에러 로그에서 정규식으로 파일명을 추출하는데, Next.js 빌드 에러 포맷이 다양해서 놓치는 경우가 많음.

**개선 방안**:
```
옵션 A: 에러 로그 전체를 AI에게 주고 "어느 파일이 문제야?" 먼저 물어보기
옵션 B: 빌드 에러 포맷별 정규식 패턴 확장 (app/, components/, lib/ 등)
옵션 C: 에러 파일 못 찾으면 → 전체 TSX 파일을 순회하며 구문 검사 (ts.transpileModule)
```

### S2. 코드 생성 품질 — Haiku의 한계

| 항목 | Haiku 결과 | 원인 |
|------|-----------|------|
| 코드 잘림 | 69줄에서 끊김 | maxTokens 8192 부족 |
| 마크다운 혼입 | F2로 대부분 제거됨 | ✅ |
| 비즈니스 로직 | 기본 CRUD만 | 모델 역량 한계 |

**권장**: E2E 테스트는 Sonnet($7.5)으로 재시도 — 코드 잘림 확률 대폭 감소 (maxTokens 16384)

### S3. 제미니 분석 참고 — 장기 로드맵

제미니가 제안한 3대 전략:
1. **Zero-Touch 프로비저닝** → ✅ Phase 4에서 이미 구현 (Supabase Management API)
2. **AI 자가 치유 루프** → ✅ F6으로 구현 (버그 수정 필요)
3. **세리온 코드 모듈화 → RAG 주입** → ❌ Phase 7 장기 과제

3번이 핵심 차별화 전략. 세리온의 실전 코드(결제, 알림톡, 예약 스케줄링)를 모듈화하여 AI 프롬프트에 주입하면, AI가 "상상"이 아닌 "조립"을 하게 됨.
