## 왜 하는가

Phase A-1에서 비주얼 에디터 기본 기능이 만들어졌다. 하지만 실전에서 터지는 치명적 문제가 있다.

### 치명적 문제: 편집할 때마다 자동 재배포 → 변경사항 날아감

현재 구조:
```
글씨 바꿈 → DB 저장 + 재배포 시작 (빌드 2분)
색상 바꿈 → DB 저장 + 재배포 시작 (빌드 2분)
이미지 바꿈 → DB 저장 + 재배포 시작 (빌드 2분)

공사 3개가 동시에 돌아감
→ 마지막에 끝난 공사가 최종 건물이 됨
→ 근데 그게 "이미지 바꾼 공사"가 아닐 수 있음
→ 사용자가 바꾼 이미지가 사라짐!
→ 비개발자: "분명 바꿨는데 왜 사라졌지?" → 버그로 느낌 → 신뢰 추락
```

## 🔴 1순위: 저장/배포 분리 + [수정사항 적용] 버튼

### 사장님 결정
- 편집할 때마다 자동 재배포 하지 마!
- 편집 → DOM 즉시 변경 + DB 저장만
- 사용자가 다 수정한 후 → [수정사항 적용] 버튼 클릭 → 그때 한 번만 재배포
- 기존 [배포] 버튼의 이름과 기능을 변경

### 변경 전 vs 변경 후

| | 변경 전 | 변경 후 |
|--|--------|--------|
| 버튼 이름 | 배포 | **수정사항 적용** |
| 편집 시 | 매번 자동 재배포 | DB 저장만, 배포 안 함 |
| 버튼 클릭 시 | 외부 공개 배포 | 수정사항 반영 재배포 |
| 10번 수정 시 | 재배포 10번 (꼬임!) | 재배포 1번 (안전!) |

### 구현 방향
- InlineEditor.tsx: onModifyComplete 호출 제거! DB 저장만 하도록 변경
- page.tsx: hasUnsavedChanges state 추가
- 인라인 편집 완료 시 hasUnsavedChanges = true
- [수정사항 적용] 버튼: hasUnsavedChanges가 true일 때 강조 표시 (뱃지 또는 색상 변경)
- 버튼 클릭 → 기존 재배포 로직 실행 → 완료 후 hasUnsavedChanges = false
- 변경사항 없으면 버튼 비활성화 또는 회색

## 🟠 2순위: 단순 치환 리스크 해결

### 문제
```
파일 안에 text-blue-500이 3군데 있으면?
→ 단순 replace → 3군데 전부 바뀜
→ 사용자는 1군데만 바꾸고 싶었음
→ Tailwind 클래스는 중복이 특히 많음
```

### 해결
클릭한 요소의 **주변 코드(context)**를 같이 보내서 유니크하게 특정

```
❌ oldText: "text-blue-500"
   → 파일 전체에서 전부 바뀜

✅ oldText: '<h1 className="text-3xl font-bold text-blue-500">'
   → 딱 그 요소만 바뀜
```

### 구현 방향
- foundry-editor.js: element-clicked 전송 시 해당 요소의 outerHTML 포함
- InlineEditor.tsx: API 호출 시 oldText에 충분한 맥락(className 전체 등) 포함
- inline-edit API: 첫 번째 매칭만 치환 (replaceAll 아닌 replace 사용)

- 변경 멈춘 후 500ms 뒤에 저장
