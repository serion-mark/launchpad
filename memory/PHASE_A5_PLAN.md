# Phase A-5: F6 뒷문 봉쇄 및 원본 보존

## 문제 상황
A-4에서 F6 부분 업데이트(머지) 로직을 넣었음에도, 인라인 편집 내용이 배포 시 날아가는 현상 발생.

**원인**: deploy.service.ts의 840~861줄 부근 'F4+F6 잘린 파일 처리' 로직 등에서 부분 업데이트(머지)를 타지 않고 전체 DB 저장을 덮어쓰는(Bypass) 뒷문이 존재함.

**비유**: 수리공(F6)이 화장실 배관(빌드 에러)만 고치면 되는데, 옛날 집 도면으로 집 전체를 다시 지어서 고객이 방금 바른 거실 벽지(인라인 편집)가 날아감.

## 현재 상태 (A-4 완료 기준)
- ✅ 비주얼 에디터 UI 완성 (클릭 → 편집 → 즉시 반영)
- ✅ DB 저장 로직 완성 (JSX 4단계 매칭)
- ✅ F6 보호 로직 (872~902줄) — freshProject 읽기 + lastModifiedFiles 보호
- ✅ syncLatestCodeToDisk (601줄) — 빌드 전 DB→디스크 동기화
- ❌ F4+F6 잘린 파일 처리 (840~861줄) — 여기가 뒷문!! 머지 안 타고 전체 저장

## 핵심 지시 (3가지)

### 【1】 F6 DB 저장 경로 전수조사
- deploy.service.ts에서 `prisma.project.update`로 `generatedCode`를 저장하는 **모든 경로** 찾기
- 특히 840~861줄 (F4+F6 잘린 파일 처리) 확인
- 각 경로가 전체 덮어쓰기인지 부분 업데이트(머지)인지 확인
- `generatedCode` 키워드로 grep해서 빠짐없이 찾을 것

### 【2】 부분 업데이트(머지) 통일
- 전수조사에서 찾은 모든 전체 덮어쓰기 경로를 머지 방식으로 변경
- 머지 방식: DB에서 최신 generatedCode 읽기 → F6이 고친 파일만 교체 → 저장
- lastModifiedFiles에 있는 파일은 절대 건드리지 않음
- A-4에서 만든 872~902줄 패턴을 그대로 적용

### 【3】 빌드 루프 내 Sync 보강
- F6 수정 후 재빌드 직전에 syncLatestCodeToDisk() 한 번 더 호출
- 현재: for 루프 밖에서 1회만 호출 (601줄)
- 변경: for 루프 안에서 매 빌드 시도 전에 호출
- 이러면 F6이 DB에 부분 저장한 후, 재빌드 전에 디스크가 DB와 100% 일치
